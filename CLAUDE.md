# Project Rules

**efremov-jmlc** — демонстрационный стенд проекта по обработке звонков отдела продаж: MinIO (S3) → Iceberg REST → Trino, Prefect для оркестрации, LiteLLM (патченый образ) и Grafana с MCP-сервером. Всё поднимается одним `docker compose`, конфигурация — через `.env`.

Деплой: push в `main` → `.github/workflows/deploy.yml` → `infra/github-runner/ci-deploy.sh` на self-hosted runner (`docker compose pull` → `build` → `up -d --remove-orphans`).

Стек: Docker Compose, Trino/Iceberg, Prefect, LiteLLM, Grafana, GitHub Actions.

## Source of truth
- CWD/.autodoc is source of truth
- Always research documentation before actions: start with ./.autodoc/index.md inside CWD or project directory, если он есть на текущей ветке, then excessively read what you may need
- Never create .autodoc at root folder — имеется в виду корень файловой системы, не корень репы: `.autodoc/` в корне данного проекта допустим и версионируется. Каталога может ещё не быть на конкретной ветке — его заводит autodoc-maintainer

## Организация доков
- `CLAUDE.md` (этот файл) — правила проекта, в git
- `CLAUDE.local.md` — личные заметки/ссылки/локальные пути, НЕ в git (уже в `.gitignore`)
- `.autodoc/` — автогенерируемый source of truth, поддерживается агентом autodoc-maintainer
- Большие сквозные флоу/пайплайны — отдельным .md; подключать в контекст только в задачах, где он нужен, не тянуть всегда

## Git
- Атомарные коммиты: один коммит = одно логическое изменение
- Никогда не коммитить .env и секреты
- `git add` только по конкретным путям; `git add -A`, `--no-verify` и plain force push заблокированы гейтом (это гейт на агента, не git-хук — человека он не держит)
- Коммитить явным pathspec: `git commit -F <файл-сообщения-вне-репы> -- путь/один.py путь/два.py`. Новые файлы сперва `git add` по тем же путям — pathspec-коммит не видит неотслеживаемое и падает. Причина правила: в репозитории параллельно работают несколько сессий с общим рабочим деревом и индексом, поэтому `git commit` без путей утаскивает чужие застейдженные файлы
- Две оговорки к нему. Pathspec-коммит берёт содержимое **рабочего дерева**, а не индекса, и переписывает индекс под него: он защищает от чужих путей, но не от чужой правки внутри твоего файла, и отменяет частичный `git add -p`. Перед коммитом посмотреть `git diff -- <те же пути>` и убедиться, что там только твои правки (если стейджил заранее — что диффа нет вовсе). При мерже и ребейзе частичный коммит запрещён (`cannot do a partial commit during a merge`) — там коммитить обычным способом
- Не склеивать git-команды в одну строку (`git add … && git commit …`): гейты проверяют строку целиком и блокируют весь вызов, поэтому первая часть не выполняется, а следующая попытка коммитит содержимое индекса, каким бы оно ни было. Отдельная команда на шаг, состояние проверять между ними
- **Рабочее дерево общее на все параллельные сессии.** Перед любой git-командой спросить себя: она трогает что-то за пределами моих путей? Если да — не выполнять. Это про `stash` (в том числе `-u`: забирает ВЕСЬ репозиторий, а не твой подкаталог), `checkout -- <путь>`, `restore`, `clean`. Прецедент: `git stash -u` в цикле проверки коммитов трижды унёс чужие незакоммиченные правки. `git reset --hard` заблокирован жёстко в `.claude/settings.json`
- Ветку не переключать и не создавать без явной просьбы пользователя: переключение меняет дерево у всех сессий разом. Изолированную проверку другой ревизии делать так, чтобы общее дерево не двигалось — `git show <sha>:путь` для чтения, отдельный временный worktree для сборки и тестов
- Прод деплоит `main`; `develop` — для разработки

## Зависимости
- Критичные зависимости — только stable и **pinned** (точные версии в lock-файле)
- Не обновляться на свежий релиз сразу: выждать, посмотреть issues (прецедент: скомпрометированный релиз litellm 1.82.8 крал креды — pinned-версии не пострадали)

## Код
- Чистый и понятный код, SOLID где уместно — без преждевременных абстракций
- Править существующее, а не создавать новое без нужды
- Минимум комментариев: комментарий только когда WHY неочевиден
- Без избыточного error handling «на всякий случай» — валидация на границах
- После правок запускать typecheck/тесты, не верить «должно работать»

## GSD / субагенты
- GSD-команды (`/gsd-plan-phase`, `/gsd-execute-phase` и т.п.) выполнять **инлайн**, без массового спавна субагентов — спавн часто вешает сессию (cold start, дублирование контекста, worktree-баги на Windows)
- Перед любым сценарием, который спавнит >1 субагента — остановиться и спросить пользователя: «инлайн или спавн?» Дефолт — инлайн
- Для мелких задач — `/gsd-quick` или `/gsd-fast` вместо полного цикла

## Документация
- После каждой новой фичи или серьёзного изменения — короткий ELI5-док простым языком: что это, зачем, как пользоваться
- Промежуточные планы и анализы — в чат, не в файлы
- Коммит с записью в changelog именовать `docs: changelog for <короткая суть пуша>`. Голое `docs: changelog` — пример из текста гейта, его брать не надо: в логе получается вереница неразличимых коммитов
- UI — на русском: лейблы, заголовки, подписи метрик, тексты для пользователя. Английские капс-лейблы вроде `TARGET RATE` переводить
- Новая документация, код, комментарии и сообщения коммитов — на английском. Это не user-facing: термин относится только к интерфейсу продукта
- Уже написанные по-русски доки не переводить: решения о миграции не было, правило задаёт язык нового
- Исключение — PR и issue: на русском. Их читают в обсуждении, а не в истории репозитория
